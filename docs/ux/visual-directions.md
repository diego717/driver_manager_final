# Direcciones Visuales para SiteOps

Este documento propone tres direcciones visuales para evolucionar SiteOps sin perder su base operativa.

Punto de partida:
- El producto ya transmite orden, seriedad y control.
- La debilidad actual no es funcional; es de personalidad visual.
- El objetivo no es "hacerlo mas lindo" de forma decorativa, sino volverlo mas reconocible y menos generico.

## Opcion A: Industrial Operativo

### Idea
Una interfaz que se sienta como una herramienta de trabajo real: precisa, robusta y sobria, con personalidad tecnica.

### Sensacion
- Control
- Fiabilidad
- Ritmo operativo
- Baja friccion

### Rasgos visuales
- Tipografia con mas caracter en headings y KPIs, manteniendo cuerpo muy legible.
- Superficies firmes, menos "card SaaS" y mas panel operativo.
- Bordes mas definidos y radios un poco mas contenidos.
- Acentos medidos en verde petroleo, oxido suave, azul señal.
- Iconografia y separadores con lenguaje tecnico, casi de consola visual refinada.

### Paleta sugerida
- Base clara: gris humo, blanco mineral, azul acero muy desaturado.
- Base oscura: grafito, pizarra, azul petroleo profundo.
- Acento principal: verde petroleo.
- Acentos secundarios: ambar operativo, azul señal, rojo alerta seco.

### Web
- Sidebar mas estructurada, con grupos bien marcados y una presencia mas utilitaria.
- Dashboard con un bloque principal dominante y secundarios mas comprimidos.
- Menos cajas equivalentes; mas contraste entre "lo urgente", "lo activo" y "lo historico".
- Estados de sync, warning y error mas expresivos sin parecer marketing.

### Android
- Pantallas con hero mas compacto y acciones primarias mas contundentes.
- Botones y chips con mas tension visual y menos aspecto de plantilla.
- Mejor jerarquia entre accion inmediata, contexto y metadata secundaria.

### Ventajas
- Es la evolucion mas natural del producto actual.
- Mantiene confianza operativa.
- Diferencia mejor a SiteOps sin romper lo ya construido.

### Riesgos
- Si se hace con timidez, puede seguir sintiendose "correcto pero comun".
- Requiere mucha precision en tipografia y jerarquia para que no quede pesado.

## Opcion B: Centro de Mision

### Idea
Tomar el lenguaje de control room moderno: mas tension, mas foco en evento, mas lectura de turno en tiempo real.

### Sensacion
- Vigilancia activa
- Prioridad
- Respuesta inmediata
- Sistema vivo

### Rasgos visuales
- Contraste mas alto.
- KPI y alertas con una presencia mucho mas dramatica.
- Layout con bloques mas tensos y algun gesto diagonal o superpuesto.
- Motion puntual en sync, alertas, cambios de estado y refresco.
- Una capa ambiental sutil que haga sentir "centro operativo", no dashboard corporativo.

### Paleta sugerida
- Base oscura dominante o claro muy contrastado.
- Verde señal, cian tecnico, ambar tactico, rojo incidente.
- Neutrales con tinte frio.

### Web
- Dashboard mas cinematografico, con un foco clarisimo en "Atencion ahora".
- Panel de sync y estado del turno mas protagonista.
- Visual principal mas memorable y con mejor sentido de pulso operativo.

### Android
- Home mas agresiva visualmente.
- Accion principal casi tipo "comenzar operacion".
- Estados mas vivos al entrar, sincronizar o retomar caso.

### Ventajas
- Mucho mas recordable.
- Le da una identidad fuerte al producto.
- Funciona muy bien si SiteOps quiere sentirse premium y especializado.

### Riesgos
- Es la opcion con mas chances de sobreactuar.
- Si nos pasamos, puede cansar en uso diario.
- Exige mucha disciplina para no caer en estetica de sci-fi generica.

## Opcion C: Editorial Tecnica

### Idea
Combinar rigor tecnico con una capa editorial refinada: menos interfaz corporativa, mas sistema pensado con criterio y gusto.

### Sensacion
- Claridad
- Autoridad
- Precision serena
- Producto maduro

### Rasgos visuales
- Tipografia mas protagonista.
- Mucho aire y mejor composicion.
- Menos decoracion, pero mas identidad en la estructura.
- Titulos y bloques con ritmo visual mas sofisticado.
- Colores mas sobrios, con un solo acento dominante.

### Paleta sugerida
- Hueso, grafito, gris humo, tinta azulada.
- Acento principal: verde oscuro o azul tecnico profundo.
- Warning/error muy limpios y secos.

### Web
- Dashboard mas limpio y jerarquico.
- Mejor lectura de secciones sin depender tanto de tarjetas repetidas.
- Sidebar, headers y tablas con presencia mas refinada.

### Android
- Pantallas mas limpias, con mejores proporciones y menos densidad visual.
- Formularios y listas con una presencia mas premium sin perder rapidez.

### Ventajas
- Muy elegante y durable.
- Baja mucho la sensacion de producto generico.
- Es facil mantener consistencia entre web, Android y desktop.

### Riesgos
- Puede quedarse demasiado contenido si queres algo mas memorable.
- Aporta prestigio, pero menos energia que las otras dos opciones.

## Recomendacion

La direccion que mejor encaja hoy con SiteOps es **Opcion A: Industrial Operativo**.

Es la que mejor conserva lo que ya funciona:
- tono serio
- foco operativo
- confianza tecnica
- continuidad entre plataformas

Y al mismo tiempo corrige lo que hoy te hace ruido:
- exceso de tarjetas similares
- jerarquia demasiado pareja
- tipografia funcional pero poco distintiva
- falta de una firma visual propia

## Mezcla Recomendada

Si queres una version mas ambiciosa sin perder equilibrio, haria esta mezcla:

- Base: Opcion A
- Intensidad visual: 20% de Opcion B
- Refinamiento tipografico: 30% de Opcion C

Traducido:
- lenguaje tecnico-industrial como base
- mas tension en dashboard y estados clave
- mejor tipografia y composicion para salir de lo generico

## Que Cambiaria Primero

Si elegimos avanzar, el orden recomendado seria:

1. Definir sistema visual corto:
   - tipografias
   - tokens de color
   - radios
   - sombras
   - reglas de panel/card
2. Rediseñar una sola pantalla ancla:
   - dashboard web
3. Pasar el mismo lenguaje a Android:
   - home
   - drivers
4. Ajustar public tracking y desktop para cerrar consistencia

## Decision Sugerida

Si queres una recomendacion directa:

- Elegi **Industrial Operativo** si priorizas coherencia, identidad y bajo riesgo.
- Elegi **Centro de Mision** si queres impacto y diferenciacion fuerte.
- Elegi **Editorial Tecnica** si queres madurez visual y una sensacion mas premium.

## Sistema Visual Propuesto

La siguiente propuesta aterriza la direccion recomendada: **Industrial Operativo**.

Objetivo:
- salir del look generico
- mantener la seriedad tecnica
- hacer que SiteOps se vea mas propio y mas seguro de si mismo

### Personalidad
- Tecnica, no fria
- Sobria, no aburrida
- Precisa, no rigida
- Operativa, no corporativa

### Firma visual
- Paneles con presencia de instrumento de trabajo
- Jerarquia fuerte entre bloque principal y secundarios
- Tipografia mas intencional en titulos, KPI y estado
- Acentos de color contenidos pero memorables
- Detalles de interfaz inspirados en paneles, estaciones y lectura de turno

## Tipografia

### Rol de fuentes
- Display y headings: `IBM Plex Sans Condensed`
- UI y cuerpo: `Source Sans 3`
- Dato tecnico puntual, labels o estados opcionales: `IBM Plex Mono`

### Regla de uso
- `IBM Plex Sans Condensed` para:
  - titulos principales
  - KPIs
  - labels de panel de alta prioridad
- `Source Sans 3` para:
  - cuerpo
  - formularios
  - tablas
  - botones
- `IBM Plex Mono` solo para:
  - timestamps
  - IDs
  - metadata tecnica
  - indicadores de sync o version

### Efecto buscado
- menos sensacion de plantilla
- mas caracter tecnico sin perder legibilidad

### Regla de distribucion
- 70% neutros
- 20% superficies y estados suaves
- 10% acentos operativos

### Ajustes clave
- menos gradientes decorativos
- neutrales con tinte tecnico, no grises puros
- warning y error secos, no chillones

## Forma y superficie

### Radios
- panel principal: `14px`
- panel secundario: `12px`
- campo y boton: `12px`
- chip y badge: `999px`

### Bordes
- usar borde visible como recurso identitario
- priorizar `1px` firme con contraste real
- reservar borde mas fuerte para foco, panel activo y prioridad

### Sombras
- cortas, densas y contenidas
- menos sensacion de SaaS flotante
- mas sensacion de pieza apoyada sobre una mesa de trabajo

### Fondo
- textura o profundidad muy sutil
- leve contaminacion azul/petroleo en neutros
- nada de efectos llamativos permanentes

## Reglas de componentes

### Paneles
- Evitar repetir el mismo peso visual en todos los paneles.
- Siempre debe existir:
  - un bloque dominante
  - dos o tres bloques de soporte
  - una capa de metadata de menor contraste

### Botones
- Primario:
  - relleno solido
  - mas peso visual
  - menos redondeado "amable"
- Secundario:
  - fondo de superficie
  - borde visible
- Danger:
  - contenido sobrio
  - rojo contenido, no brillante

### Chips y estados
- usar chips solo si agregan lectura rapida
- estados criticos con mejor contraste y menor decoracion
- sync, SLA y prioridad deben verse como sistema, no como badges aislados

### Tablas y listas
- dar mas aire horizontal
- reforzar encabezados y labels
- separar accion, entidad y metadata con mas claridad

## Motion

### Si
- pulso sutil en sync
- entrada corta en KPI y bloque de atencion
- hover con desplazamiento muy corto
- transicion clara al cambiar estado

### No
- rebotes
- animacion constante
- efectos de marketing

## Aplicacion al Dashboard Web

### Intencion
La home no debe sentirse como un tablero generico con metricas. Debe sentirse como el inicio del turno.

### Estructura sugerida

#### Header
- Titulo fuerte con `IBM Plex Sans Condensed`
- subtitulo corto, operacional
- estado de sincronizacion visible y elegante
- una CTA primaria real

Ejemplo:
- titulo: `Centro del turno`
- subtitulo: `Prioriza incidencias, registros activos y desvio de SLA.`

#### Banda superior
- izquierda:
  - KPI principal de incidencias criticas abiertas
- derecha:
  - registros en curso
  - fuera de SLA
  - ultima sincronizacion

Regla:
- el KPI principal debe verse 2x mas importante que los secundarios

#### Visual principal
- un solo grafico ancho
- barras con linea objetivo
- selector `24h / 7d`
- contexto y lectura lateral mas compacta

#### Bloque "Atencion ahora"
- lista de 5 items accionables
- cada item con:
  - prioridad
  - cliente o equipo
  - estado
  - tiempo en estado
  - CTA directo

Regla:
- debe competir visualmente con el grafico, no quedar como bloque secundario timido

### Comportamiento visual
- sidebar mas utilitaria
- menos glass y menos glow
- paneles con borde, contraste y titulo mas seco
- estados importantes con acento lateral o bloque de tono suave

## Aplicacion a Android

### Home
- Hero mas corto
- accion principal mas decidida
- menos texto introductorio
- mas foco en "que hago ahora"

### Drivers
- separar claramente:
  - carga
  - lista
  - accion por item
- hacer que el upload se sienta como herramienta, no como formulario generico

### Lenguaje visual
- cards menos blandas
- mejor peso de labels
- uso de monospace en version, archivo, fecha o sync cuando aporte lectura tecnica

## Antes y Despues Esperado

### Antes
- correcto
- limpio
- consistente
- algo generico

### Despues
- tecnico
- reconocible
- mas confiado
- mas propio

## Handoff de Implementacion

Si decidimos aplicarlo en codigo, la secuencia recomendada es:

1. Actualizar tokens base en:
   - `dashboard.css`
   - `mobile-app/src/theme/palette.ts`
2. Introducir jerarquia tipografica nueva en:
   - dashboard web
   - home Android
   - drivers Android
3. Redibujar la pantalla ancla:
   - dashboard web
4. Expandir el lenguaje a:
   - Android
   - public tracking
   - desktop

## Recomendacion Practica

El mejor siguiente paso no es cambiar todo a la vez.

Lo haria asi:
- definir los tokens y reglas del sistema visual
- aplicar la direccion al `dashboard` web
- usar ese resultado como referencia para Android y desktop

Eso te deja ver rapido si la nueva personalidad funciona sin arriesgar toda la interfaz al mismo tiempo.
